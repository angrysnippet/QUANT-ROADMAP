## Why today matters
Beginner: *idea → code → measure complexity.* Expert: *constraints → target complexity → candidate algorithms → code.* The input size often names the algorithm family before you've solved anything.

---

## Block 1 — The constraint-first rule
A rough table every quant dev internalises (≈10⁸–10⁹ simple ops/second):

| n | Feasible complexity |
|---|---------------------|
| ≤ 20 | exponential `2ⁿ`, `n!` |
| ≤ 500 | `O(n³)` |
| ≤ 5,000 | `O(n²)` |
| ≤ 10⁶ | `O(n log n)` |
| ≤ 10⁸ | `O(n)` / streaming |

At `n = 100000`, `O(n²)` = 10¹⁰ ops → too slow. **The constraint eliminates whole families before you code.**

---

## Block 2 — DSA · Designing backward from the bound
Given `n = 2·10⁵` and a 1-second limit, you need ~`O(n log n)`. That immediately suggests **sorting, binary search, heaps, or a single DP pass** — and rules out nested loops. Pick the data structure that *hits the target complexity.*

**Task:** for three constraints (`n=18`, `n=2000`, `n=10⁶`), name the complexity you'd aim for and one technique each.

---

## Block 3 — DSA · Reading the limits like clues
- `n ≤ 20` → subsets/bitmask DP (`2ⁿ`).
- "array up to 10⁵, answer over subarrays" → O(n) or O(n log n) → prefix sums / sliding window / two pointers.
- "10⁶ queries" → O(1)/O(log n) per query → hashing, precomputation, segment tree.

---

## Block 4 — Mathematics (Growth rates)
`log n` ≪ `√n` ≪ `n` ≪ `n log n` ≪ `n²` ≪ `2ⁿ` ≪ `n!`. **Exercise:** at 10⁸ ops/sec, how long is `n²` for `n=10⁵`? (~100s — infeasible.) **Challenge:** why does `O(n log n)` feel "basically linear" in practice? (log n ≈ 17 even at 10⁵.)

## Block 5 — Quant Thinking (Complexity *is* money and latency)

This is not academic for a quant:
1. **Backtesting:** a strategy over 5 years of 1-minute bars ≈ 2M rows; over tick data, *billions*. An `O(n²)` "for each trade, scan all prior trades" backtest is dead on arrival — you need `O(n)` streaming with rolling state.
2. **HFT latency budgets:** a tick-to-trade path may have a **microsecond** budget. Complexity per event, not asymptotics alone, decides whether you're in the market.
3. **Vectorization:** a Python `for` loop over a 10⁶-row price series can be 100× slower than the same logic in `numpy`/`pandas` (C-level, SIMD). Same `O(n)`, vastly different constant.

**Hard puzzle:** you must compute a rolling 50-period moving average over 10⁷ ticks, updated live. Naively recomputing the window each tick is `O(n·k)`. What's the `O(n)` design? (Maintain a running sum: add new, subtract the one leaving — Day 60's sliding window.)

**Career connection:** "what's the complexity, and what's the constant factor?" is asked constantly in quant-dev interviews — and it directly sets your capacity, cost, and edge.

## Quant Thinking Track — Feasibility Before Effort
