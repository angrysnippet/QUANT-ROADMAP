## Why today matters
"Delhi → Mumbai, shortest route?" → Dijkstra (easy). But "shortest route **AND** budget < ₹5000" — the constraint changes the problem; shortest path alone isn't enough.

---

## Block 1 — Problem Decomposition
Cities + roads + travel cost + budget limit → object **Graph**, goal **Minimum distance**, constraint **Budget**. So: *Graph + optimization + constraint.* The decomposition matters more than the final algorithm.

---

## Block 2 — DSA · Shortest Path Refresher
`A--5-->B, A--2-->C, C--1-->B`: shortest A→B is `A→C→B = 3`, not 5 — classic **Dijkstra.** Signal: *minimum cost / distance / fastest / cheapest* → shortest path.

---

## Block 3 — DSA · When Constraints Appear
"Shortest path **using at most K edges**" — plain Dijkstra isn't enough; you need extra state: `(node, edges_used)`.

**Huge lesson:** *constraints often create new state variables.* Airline "cheapest flight with at most 2 stops" → state `(city, stops_used)`, not `city` alone.

---

## Block 4 — Mathematics (State expansion)
Normal Dijkstra: state `(node)`. Constrained: `(node, constraint_info)` — e.g. `(node, fuel)`, `(node, stops)`, `(node, budget)`, `(node, time)`.

**Deep insight:** many advanced problems = **classic algorithm + extra state.**

**Exercise:** "shortest path with at most 3 flights" — what state would you track?

---

## Block 5 — Quant Thinking (Reality has constraints)

Textbook: shortest route. Reality: shortest route *within budget, within time, within risk limit.* The real world always adds constraints.

**Examples:** portfolio optimization (max return, risk < threshold); trading infra (lowest latency, cost limit).

**Hard puzzle:** become a quant researcher = max skill growth subject to limited time/energy/money — why is this harder than simple optimization? (Multiple coupled constraints.)

**Career connection:** portfolio allocation (return + risk), execution (speed + cost), research planning (impact + time) — strong researchers model **objective AND constraints** together.
