## Why today matters
Students: *problem → find algorithm → solve.* Researchers: *problem → model reality → discover structure → create algorithm.* The second is far more powerful.

---

## Block 1 — The secret behind every algorithm
Each major algorithm began as an **observation** of structure: Binary Search ("answers form a boundary"), DP ("subproblems repeat"), Dijkstra ("cheapest state first can't get worse"), Greedy ("the local choice is safe"). *Algorithms are born from observations, not code.*

*Why it matters:* if you can spot structure, you can design a method even for problems with no named algorithm.

---

## Block 2 — DSA · The Algorithm Design Framework
For any new problem ask: (1) **state?** (city / cell / portfolio config) (2) **transitions?** (move / trade / study / buy / sell) (3) **objective?** (min / max / count / reachability) (4) **constraints?** (budget / time / capacity / dependencies). Most algorithms emerge naturally after these.

---

## Block 3 — DSA · Invention exercise
"Robot navigation, battery-limited, recharge stations exist." Don't ask *which chapter?* — ask: **state** `(position, battery)`, **transitions** move/recharge, **goal** reach destination. The structure becomes visible.

**Deep insight:** strong solvers don't search memory — they **search structure.**

---

## Block 4 — Mathematics (Abstraction)
Portfolio optimization (*items / value / risk / budget*) → Knapsack. Research planning (*tasks / dependencies / rewards*) → Graph DP.

**Exercise:** why is abstraction often harder than implementation? **Challenge:** when abstracting, what information must be *removed*, and what must *remain*?

---

## Block 5 — Quant Thinking (Researchers think in models)

Weak: *problem → code.* Strong: *problem → model → algorithm → code.* The model determines everything. Facing thousands of stocks, a researcher asks *what variables matter?*, not *what library?*

**Hard puzzle:** model "your quant journey" — which state variables matter (DSA skill, Probability skill, Python skill, project experience), and which don't? That's exactly what model-building looks like.

**Career connection:** the highest-paid technical people are **model builders**, not code writers — *correct model → correct algorithm → correct solution*; a wrong model ruins everything.

## Quant Thinking Track — Algorithms Are Discovered, Not Memorised
Many believe smart people *know more algorithms.* Often the truth: they **see structure faster.** Once state / transitions / goal / constraints are visible, the algorithm is far easier to identify.
