## Block 1 — The memory hierarchy (the central fact)
Modern CPUs are *fast*; memory is *slow* — and the gap is enormous. The hierarchy, by rough access latency:
- **registers** — <1 ns (instant),
- **L1 cache** — ~1 ns (~4 cycles), tiny (~32 KB),
- **L2 cache** — ~4 ns, small (~256 KB–1 MB),
- **L3 cache** — ~15 ns, larger (~8–32 MB, shared),
- **main memory (RAM)** — **~100 ns** — *100× slower than L1*,
- (disk/network — microseconds to milliseconds — astronomically slower).

The killer fact: **a cache miss (going to RAM) costs ~100 ns — enough time for the CPU to execute *hundreds* of instructions.** So a program that *waits on memory* wastes the CPU's enormous speed. **Modern performance is dominated by *memory access patterns*, not arithmetic** — the CPU can compute far faster than it can fetch data. This inverts the naive "count the operations" view of performance.

*Why it matters:* if you don't think about cache, your beautifully-optimized algorithm spends most of its time *waiting for memory* — and you'll never know without profiling. Cache-awareness is the highest-leverage optimization on modern hardware.

---

## Block 2 — Cache lines and locality
The CPU doesn't fetch one byte at a time — it fetches a **cache line** (typically **64 bytes**) at once. This makes two kinds of *locality* crucial:
- **spatial locality** — if you access data, you'll likely soon access *nearby* data; the cache line brings neighbors along for free. So **accessing memory sequentially is fast** (each cache line serves ~8 doubles before the next fetch),
- **temporal locality** — recently-accessed data is still in cache; reusing it is fast.

The practical consequences:
- **sequential access ≈ cache-friendly** (you use the whole 64-byte line, hardware *prefetches* the next) → fast,
- **random access ≈ cache-hostile** (every access may miss, fetching a full line to use 8 bytes of it) → slow,
- **contiguous data structures (arrays/vectors) win** because they guarantee spatial locality; **pointer-chasing structures (linked lists, trees, hash maps with chaining) lose** because each node may be anywhere in memory → cache miss per hop.

This is *why* `std::vector` usually beats `std::list` *even for operations where the list has better big-O* — the vector's contiguity makes it cache-friendly, and cache dominates.

**Concept check:** summing 10 million integers stored in a contiguous `std::vector` vs the same integers in a `std::list` (linked list). The list has the same `O(n)` complexity — why is it dramatically slower? (**Cache behavior.** The `vector` is *contiguous*: accessing it sequentially uses each 64-byte cache line fully (~16 ints per line) and the hardware *prefetches* ahead, so memory access is nearly free — it's cache-friendly. The `list` is *pointer-chained*: each node is a separate heap allocation that could be *anywhere* in memory, so traversing it causes a **cache miss per node** (~100 ns each), and there's no prefetching (the CPU can't predict where the next node is). Same big-O (`O(n)`), but the vector might be **5–50× faster** purely due to cache locality. This is *the* canonical demonstration that **memory layout dominates big-O** for realistic data — and why data-oriented design prefers contiguous arrays.)

## Block 3 — Data-Oriented Design (DOD)
The performance philosophy that follows: **design your data layout around how it's *accessed*, not around abstract objects.** A key technique is **Struct-of-Arrays (SoA)** vs **Array-of-Structs (AoS)**:
- **Array-of-Structs (AoS)** — the "natural" OOP layout: an array of `Order` objects, each holding `{price, size, id, timestamp, ...}`. To sum all *prices*, you stride through memory, pulling whole 64-byte objects but using only the 8-byte price — wasting ~⅞ of each cache line,
- **Struct-of-Arrays (SoA)** — separate arrays: `prices[]`, `sizes[]`, `ids[]`. To sum all prices, you read `prices[]` *contiguously* — every byte of every cache line is used, and it *vectorizes* (SIMD, Day 227).

For operations that touch *one field across many objects* (extremely common in quant: "compute the VWAP over all trades," "find all orders above a price"), **SoA is dramatically faster** because it's cache-friendly and SIMD-friendly. This is the core insight of data-oriented design (championed in game engines and HFT): **lay out data for the access pattern**, even if it's less "object-oriented." It's a deliberate departure from textbook OOP in service of the machine.

## Block 4 — Mathematics · the cost model and why big-O misleads
The classical algorithmic cost model (count operations, big-O) *assumes uniform memory access cost* — every access is "O(1)." **This assumption is false on modern hardware**, where access cost varies by ~100× depending on cache. So the *real* cost model must account for memory:
```
real cost ≈ (computation) + (cache misses × ~100 ns)
```
And for memory-bound code (most data processing), the **second term dominates**. Consequences that overturn naive intuition:
- **a cache-friendly `O(n)` scan can beat a cache-hostile `O(log n)` lookup** for realistic `n` — e.g., a *linear scan of a sorted array* often beats a *binary search tree* because the array is contiguous (cache-friendly, prefetched) while the tree is pointer-chasing (cache miss per level). For `n` up to thousands, *linear-but-contiguous* frequently wins,
- **the constant factors hidden by big-O are *huge* and memory-driven** — big-O tells you *asymptotic scaling*, but for the *finite, cache-sensitive* sizes of real systems, the constants (set largely by memory behavior) often decide the winner,
- **algorithmic complexity and cache behavior are *both* real, and you must consider both** — a galaxy-brained `O(log n)` algorithm with terrible locality can lose to a "dumb" `O(n)` scan with perfect locality. The right answer is *empirical* (profile, Day 229), but the *intuition* must include the memory hierarchy.

The deep lesson: **big-O is a *necessary but insufficient* model of performance — it captures algorithmic scaling but ignores the memory hierarchy that dominates modern hardware.** A complete performance analysis considers *both* the algorithm's complexity *and* its memory access pattern (cache behavior, locality, allocation). This is *why* the performance mindset (Day 221) requires *mechanical sympathy* — you can't predict real performance from big-O alone; you must understand what the *memory system* does. The mathematical model of computation that ignores the memory hierarchy (the RAM model) is a *useful abstraction for algorithm design* but a *misleading guide to real-world performance* — and bridging that gap is the essence of high-performance engineering.

## Block 5 — Quant Thinking · design for the machine, measure don't guess
Memory and cache crystallize two of the most important principles of high-performance systems engineering — principles that separate a researcher who *uses* computers from an engineer who *commands* them. **First: data layout is a first-class design decision, and on modern hardware, *how you arrange data in memory often matters more than the algorithm you run on it.*** This is deeply counterintuitive coming from algorithms courses (which teach you to optimize big-O) and from research Python (where memory is abstracted away), but it's *the* central fact of modern performance: the CPU is so fast and memory so (relatively) slow that **memory access patterns dominate runtime**, so designing data for *locality* (contiguous, cache-friendly, SoA where appropriate) is the highest-leverage optimization. The mature systems engineer thinks about data layout *before* algorithms for performance-critical code — "how will this data be accessed, and how do I lay it out so the cache is happy?" This is *data-oriented design*, and it's why HFT systems, game engines, and high-performance databases all obsess over memory layout. In quant systems specifically — processing millions of market-data ticks, computing signals over large arrays, running backtests over years of data — *cache efficiency directly determines throughput and latency*, which determines how many ideas you can test (research velocity) and how fast you react to the market (execution edge, Day 186). **Second, and equally important: measure, don't guess.** Performance intuition is *notoriously* unreliable — even experts are routinely wrong about where the bottleneck is, because modern hardware (caches, branch predictors, out-of-order execution, prefetchers) is too complex to reason about perfectly. The cardinal rule of optimization is **profile first** (Day 229): *measure* where time is actually spent, optimize *that*, and *measure again* to confirm the improvement. The corollary is **don't prematurely optimize** — write clear, correct code first, profile to find the *actual* hot spots (usually a small fraction of the code — the 80/20 rule), and optimize *those* with data-oriented techniques, leaving the rest readable. This disciplined, *empirical* approach to performance — *measure, optimize the real bottleneck, measure again* — is the same intellectual honesty that runs through the whole curriculum (don't trust your backtest, Day 199; don't trust your intuition, *measure*): **in performance, as in research, the data (the profiler) is the authority, not your intuition.** The unifying wisdom: **understand the machine (mechanical sympathy), design data for how the machine accesses it (data-oriented design, cache-awareness), and let measurement — not intuition — drive optimization.** The quant developer who internalizes this builds systems that are *genuinely* fast (not just "should be fast"), tests more ideas per day (research velocity), and reacts faster than competitors (execution alpha) — turning the abstract theory of the nine pillars into *deployed systems that win in the real, hardware-constrained, latency-sensitive world.* Performance is not magic; it's understanding the machine and respecting the measurement.

**Hard puzzle:** you're building a backtester that processes 10 years of tick data (billions of ticks) and computes a moving-average signal over them. Your first version stores each tick as an `Order` object `{double price; long timestamp; int size; char side; ...}` in a `std::vector<Order>` and is frustratingly slow. Diagnose the likely performance problem and redesign for speed using this day's concepts. (**Diagnosis — it's almost certainly *memory-bound*, wasting most cache bandwidth:** (1) **Array-of-Structs (AoS) cache waste** — your moving-average computation only needs the `price` field, but storing ticks as `Order` *objects* means the prices are *strided* in memory (separated by the full size of each `Order` struct — maybe 32–64 bytes). When you scan to sum prices, each 64-byte cache line you fetch contains *only one or two prices* (the rest is timestamp, size, side, etc. that you don't need for this computation) — so you're using maybe 1/8 of every cache line, wasting ~⅞ of your memory bandwidth and causing far more cache misses than necessary (Block 3). For billions of ticks, this is *catastrophic* — you're memory-bound and most of your fetched data is useless for the SMA. (2) **No vectorization** — strided access also prevents the compiler from using SIMD (Day 227) effectively. **Redesign for speed (data-oriented design):** (1) **Switch to Struct-of-Arrays (SoA)** — store the data as *separate contiguous arrays*: `std::vector<double> prices; std::vector<long> timestamps; std::vector<int> sizes;` etc. Now the moving-average computation reads `prices[]` *fully contiguously* — every byte of every cache line is a price you need, maximizing cache-line utilization (~8 prices per line instead of 1–2), enabling hardware prefetching, and allowing the compiler to *vectorize* the sum with SIMD. This alone could give a **5–10×+ speedup** for the price-scanning computation. (2) **Ensure contiguous storage** — `std::vector` (contiguous) not `std::list` or pointer-based structures (Block 2). (3) **Process in cache-friendly order** — scan sequentially (forward), which the prefetcher loves, rather than random access. (4) **Consider the working set** — for a rolling moving average, you only need a window of recent prices; structure the computation to keep the active window in cache (temporal locality), e.g. a rolling sum updated incrementally (`O(1)` per step) rather than re-summing the window each time (which is *also* an algorithmic improvement — `O(n)` total instead of `O(n×window)`). (5) **Profile to confirm** (Day 229) — use `perf stat` to measure cache-miss rate before/after; the SoA version should show *dramatically* fewer cache misses. **The deep lesson:** the slow version wasn't slow because of a bad *algorithm* (a moving average is simple) — it was slow because of a bad *data layout* that made it *memory-bound* and cache-hostile. The fix is **data-oriented design**: lay out the data (SoA, contiguous) for the *access pattern* (sequential scan of prices), turning a memory-bound disaster into a cache-friendly, vectorizable, fast computation. This is *exactly* the "memory layout matters more than the algorithm" lesson (Day 221/222), and it's *pervasive* in quant systems — backtesters, market-data handlers, and signal engines all process huge arrays where cache efficiency *is* performance. A quant developer who knows AoS-vs-SoA and cache-friendly layout builds backtesters that run 10× faster — meaning 10× more ideas tested per day, a direct research-velocity (and thus alpha-discovery) advantage. The naive OOP "an array of Order objects" feels natural but fights the machine; the data-oriented "separate arrays of fields" works *with* the machine. Design data for the cache, measure to confirm, and the same hardware does far more work.)

**Career connection:** "what's the most important thing to understand for high-performance C++?" Strong answer: **the memory hierarchy and cache behavior** — on modern hardware, **memory access, not computation, is usually the bottleneck** (a cache miss to RAM costs ~100 ns, enough for *hundreds* of instructions), so **how you lay out data in memory often matters more than your algorithm's big-O.** The key facts: the CPU fetches **64-byte cache lines**, so **sequential/contiguous access is fast** (full cache-line use + hardware prefetching) while **random/pointer-chasing access is slow** (cache miss per hop) — which is why `std::vector` (contiguous) routinely beats `std::list` (pointer-chained) *even with the same big-O*. The design philosophy is **data-oriented design**: lay out data for the *access pattern*, e.g. **Struct-of-Arrays** (separate `prices[]`, `sizes[]` arrays) instead of **Array-of-Structs** (array of `Order` objects) when you process one field across many items — it maximizes cache-line utilization and enables SIMD. And the cardinal rule: **profile, don't guess** — measure where time is actually spent (cache misses included) and optimize the real bottleneck. In quant systems (backtesters, market-data handlers, signal engines processing billions of ticks), cache efficiency directly determines throughput and latency — and thus research velocity and execution edge. Naming the memory hierarchy, cache lines, contiguous-vs-pointer-chasing, AoS-vs-SoA/data-oriented design, and profile-don't-guess is the high-performance-systems answer.

## Block 6 — C++ (systems) · Contiguous vs pointer-chasing, AoS vs SoA
```cpp
#include <vector>
#include <list>
#include <chrono>
#include <iostream>

template <typename F> long time_us(F f) {
    auto s = std::chrono::high_resolution_clock::now(); f();
    auto e = std::chrono::high_resolution_clock::now();
    return std::chrono::duration_cast<std::chrono::microseconds>(e - s).count();
}

int main() {
    const int N = 10'000'000;
    // contiguous vector vs pointer-chasing list — SAME O(n), very different cache behavior
    std::vector<double> vec(N, 1.0);
    std::list<double>   lst(N, 1.0);
    volatile double sink = 0;
    std::cout << "vector sum: " << time_us([&]{ double s=0; for(double x:vec) s+=x; sink=s; })
              << " us\n";
    std::cout << "list   sum: " << time_us([&]{ double s=0; for(double x:lst) s+=x; sink=s; })
              << " us  <- pointer-chasing = cache misses, much slower\n";

    // AoS vs SoA: summing one field across many records
    struct Order { double price; long ts; int size; char side; };
    std::vector<Order> aos(N, {1.0, 0, 0, 'B'});
    std::vector<double> soa_price(N, 1.0);                 // just the prices, contiguous
    std::cout << "AoS price-sum: " << time_us([&]{ double s=0; for(auto&o:aos) s+=o.price; sink=s; })
              << " us  (wastes cache lines on unused fields)\n";
    std::cout << "SoA price-sum: " << time_us([&]{ double s=0; for(double p:soa_price) s+=p; sink=s; })
              << " us  <- contiguous prices, cache-friendly & vectorizable\n";
}
```
The `vector` crushes the `list` (contiguous vs pointer-chasing) and **SoA** beats **AoS** (full cache-line use vs wasted bytes) — *identical* big-O, dramatically different real performance, all from memory layout.

**Linux:** `g++ -O3 -march=native cache.cpp -o cache && perf stat -e cache-misses ./cache` — `perf` reveals the cache-miss counts driving the gap; the list and AoS versions will show far more misses.
