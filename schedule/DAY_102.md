## Block 1 — C++ (From array to answer space)
Normal binary search finds an index in a sorted array. Today we search the **answer space** itself — e.g. possible capacities `1..10` — not an array.

*Why it matters:* whenever the optimum is hard but "does X work?" is easy *and monotonic*, binary search applies even with no array in sight.

---

## Block 2 — DSA (The core pattern)
"Assign books to students, minimise the maximum pages" — hard directly. Transform *minimum answer* → *can answer X work?* Define `check(X)` → true if feasible. Then binary-search X.

---

## Block 3 — DSA (Monotonic property — the key requirement)
```
check(50)=F  check(60)=F  check(70)=F  check(80)=T  check(90)=T  → FFF TTT
```
There's a single boundary — **monotonicity** — which is exactly what binary search exploits.

**Signal:** small answer fails, large answer works (or vice versa) → binary search on answer.

## Block 5 — Mathematics (Search-space reduction)
Answer in `1..1,000,000`: linear = 10⁶ checks; binary = `log₂(10⁶) ≈ 20`.

**Exercise:** steps for `1..1024`? (10.) **Challenge:** why is binary search fundamentally divide-and-conquer?

## Block 6 — Quant Thinking (Feasibility before optimization)

Weak thinker asks *what's best?* immediately; strong thinker asks *can this work?* first. "Can target return R be hit under the risk limit?" → check true/false → search over R.

**Hard puzzle:** 100 study hours, goal skill level X — can you define `check(X)`?

**Career connection:** resource allocation, trading systems, scheduling, operations research, capacity planning are secretly *optimization → feasibility → binary search.*
