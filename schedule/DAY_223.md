## Block 1 — What templates are
A **template** is a *blueprint* for code, parameterized by type (or value). You write the logic *once*, and the compiler **generates a specialized version for each type you use it with**, at *compile time*:
```cpp
template <typename T>
T max_value(T a, T b) { return a > b ? a : b; }
// max_value(3, 5) generates an int version; max_value(2.1, 3.4) a double version — each fully optimized
```
The key: **the specialization happens at compile time**, so each generated version is *as fast as if you'd hand-written it for that type* — no runtime cost, no boxing, no dispatch. This is **compile-time (static) polymorphism** — "polymorphism" (one interface, many types) resolved *when compiling*, not when running. Templates are *the* mechanism behind C++'s zero-cost abstraction (Day 221).

*Why it matters:* templates let you write *reusable, generic* code (one algorithm for all numeric types) that's *as fast as specialized code* — exactly what you need for performance-critical quant libraries (a pricer that works for `double`, `float`, or a custom type, with no speed penalty).

## Block 2 — The STL: templates in action
The **Standard Template Library (STL)** is the crown jewel of generic programming — and what you'll use constantly:
- **containers** — `std::vector` (Day 222, your default), `std::array`, `std::map`/`std::unordered_map`, `std::deque` — all templated on element type, so one implementation serves all types,
- **algorithms** — `std::sort`, `std::find`, `std::accumulate`, `std::transform`, `std::lower_bound` — templated, working on *any* container via iterators,
- **iterators** — the generic "pointer-like" interface that connects algorithms to containers (any algorithm works on any container that provides the right iterators),
- **utilities** — `std::pair`, `std::tuple`, `std::optional`, `std::function`.

The STL embodies generic programming's power: `std::sort` works on a `vector<double>`, a `vector<Order>`, or any type with a comparison — *one* implementation, *zero* overhead (the comparison inlines), often *faster* than C's `qsort` (which uses a function pointer that can't inline). **Learn the STL deeply** — it's the productive, fast, correct foundation of all modern C++, and reinventing its wheels (your own sort, your own hash map) is almost always slower and buggier.

## Block 3 — Templates vs virtual functions (static vs dynamic polymorphism)
C++ has *two* kinds of polymorphism, and choosing between them is a key design decision:
- **Templates (compile-time / static polymorphism):** the type is known at *compile time*, so the compiler generates specialized, *inlinable*, zero-overhead code. Fast, but: code bloat (a copy per type), longer compile times, harder error messages, and the types must be known at compile time.
- **Virtual functions (runtime / dynamic polymorphism):** a base class with `virtual` methods; the actual type is resolved at *runtime* via a **vtable** (a table of function pointers). Flexible (handle types unknown at compile time, store mixed types in one container), but: a *runtime cost* (an indirect call through the vtable — can't inline, may cause a cache miss and branch misprediction).

The trade-off:
- **use templates** when the type is known at compile time and performance matters (the hot path) — zero overhead,
- **use virtual functions** when you need *runtime* flexibility (e.g. a `std::vector<std::unique_ptr<Strategy>>` holding different strategy types decided at runtime) and the (small) virtual-call cost is acceptable.

In *performance-critical* code (the hot path of a trading system), prefer templates to avoid the virtual-call overhead; in *flexible architecture* code (plugin strategies, configuration), virtual functions are fine and cleaner.

**Concept check:** in the microsecond-latency hot path of an order-matching engine, would you dispatch via a `virtual` function call or a template? (**Template** — a virtual call is an *indirect* call through the vtable that the compiler *cannot inline*, costs a pointer dereference, and risks a branch misprediction and cache miss — adding nanoseconds *per call*, which in a hot path called millions of times adds up to real latency. A template resolves the call at *compile time*, allowing the compiler to *inline* it (zero call overhead) and optimize across the boundary. On the hot path, you want compile-time (static) polymorphism. *Off* the hot path — say, choosing which strategy object to run once per session — virtual functions are perfectly fine and give cleaner, more flexible code. Match the polymorphism to the performance requirement.)

## Block 4 — Mathematics · compile-time computation and modern generic features
Templates are more than generic types — they enable *computation at compile time*, shifting work from runtime to the compiler:
- **template metaprogramming** — historically, templates were (ab)used to compute things at *compile time* (the compiler runs a mini-program during compilation), so the runtime cost is *zero* (the answer is baked into the binary). Modern C++ has cleaner tools for this:
- **`constexpr`** (C++11/14/17) — functions and variables computed at *compile time*: `constexpr double pi = 3.14159...;` or a `constexpr` factorial computed by the compiler. The result is a *constant* in the binary — zero runtime cost. Hugely useful for lookup tables, compile-time configuration, and computations whose inputs are known at compile time,
- **concepts** (C++20) — *constrain* template type parameters with named requirements (e.g. `template <std::floating_point T>`), giving *vastly* better error messages and clearer interfaces (the historic pain of templates was incomprehensible error messages; concepts fix this),
- **`auto` and template argument deduction** — the compiler infers types, reducing verbosity,
- **variadic templates** — templates with *any number* of arguments (how `std::tuple`, `printf`-style functions, and `std::make_unique` work),
- **SFINAE / `if constexpr`** — compile-time branching: `if constexpr (condition)` compiles *only* the taken branch, enabling type-dependent logic with zero runtime cost.

The unifying mathematical idea: **templates and `constexpr` move computation from *runtime* to *compile time*** — work the compiler does *once* (when building) costs *nothing* at runtime (when trading). For latency-critical systems, this is gold: precompute lookup tables, specialize code per type, eliminate runtime branches — all *for free* at runtime, paid for once at compile time. It's the ultimate expression of "do the work ahead of time so the hot path is fast." This is a recurring performance theme — *precompute what you can* — elevated to a language feature: the compiler becomes a powerful precomputation engine, and generic programming lets you write *one* expressive source that compiles to *many* optimal specialized versions.

## Block 5 — Quant Thinking · generic, reusable, AND fast — and computation as compile-time
Templates and generic programming embody a principle that resolves a tension most languages can't: **you can have abstraction, reusability, *and* maximal performance simultaneously — if the abstraction is resolved at compile time.** In most languages, generic/reusable code costs performance (Java generics box and dispatch; Python's duck typing interprets; runtime polymorphism adds indirection). C++ templates break this trade-off: **write generic code once, and the compiler generates specialized, fully-optimized, zero-overhead versions for each concrete type** — you get the *productivity* of "write once, works for all types" *and* the *speed* of hand-written specialized code, with no compromise. This is the deep reason C++ can build *both* expressive libraries (the STL — generic containers and algorithms you use everywhere) *and* the fastest systems on earth (HFT engines, exchanges): the genericity compiles *away*. For the quant developer, this means you can build a *reusable, well-abstracted* quant library — a pricer templated on the numeric type, a generic backtester that works on any signal, a container that holds any order type — *without* sacrificing the microsecond performance the hot path demands. You don't have to choose between "clean, reusable code" and "fast code"; templates give you both. This connects to the curriculum's recurring lesson about *the right abstraction* (state-space unifying time-series models, Day 203; factor models structuring risk, Day 211) — now realized in *code*: a good generic abstraction (a template) captures the *shared structure* once and specializes optimally for each case, just as a good mathematical framework captures the general case and specializes to the particulars. And the **compile-time computation** angle (Block 4) reveals a powerful systems principle: **move work from runtime to compile time wherever possible** — `constexpr` precomputation, template specialization, and compile-time branching shift effort from the *latency-critical hot path* (where every nanosecond costs money in trading) to the *compiler* (where the cost is paid once, at build time, and is invisible at runtime). This is the code-level version of "precompute what you can" — a theme that recurs throughout performance engineering (cache the discount curve, Day 161; precompute lookup tables; amortize setup cost). The mature quant developer thinks not just "what does this code do?" but "*when* does it do it — compile time or runtime? — and can I shift more of it to compile time to make the hot path faster?" The unifying wisdom: **great C++ achieves genericity, reusability, *and* performance by resolving abstractions at compile time — write expressive, reusable generic code that compiles to optimal specialized machine code, and push computation to compile time so the runtime hot path is as lean and fast as possible.** This is the engineering embodiment of zero-cost abstraction (Day 221), and it's *why* the world's most performance-critical *and* most heavily-engineered systems (trading infrastructure) are built in C++: it uniquely lets you have the abstraction *and* the speed, the reusability *and* the microseconds. Master templates, and you can build quant libraries that are *both* a joy to use *and* fast enough to trade on.

**Hard puzzle:** you're designing a quant library with a function to compute the present value of a cash-flow stream (Day 161). It should work for `double` precision, `float` (for speed/memory in some contexts), and potentially a custom high-precision or fixed-point type — *and* it must be fast on the hot path. Should you use templates, virtual functions, or function overloading, and what are the trade-offs? (**Use templates** — this is the textbook case for compile-time generic programming, and the reasoning illuminates the whole design space: **Why templates win here:** (1) **The types are known at compile time** — you know *at the call site* whether you're computing in `double`, `float`, or a custom type; there's no need for *runtime* type flexibility. So you want the type resolved at *compile time* (static polymorphism), not runtime. (2) **Performance on the hot path** — present-value/discounting is a *hot-path numerical computation* (called constantly in pricing and risk); a template generates *specialized, inlinable, zero-overhead* code for each type (`double` version uses double instructions, `float` version uses float instructions — each optimal), with no runtime dispatch cost. A `double` PV and a `float` PV are *different optimized machine code*, exactly as if hand-written. (3) **Write once, works for all numeric types** — `template <typename T> T present_value(const std::vector<T>& cashflows, T rate)` — one implementation serves `double`, `float`, and the custom type, with the compiler generating each specialization; adding a new numeric type requires *no* new code. (4) **Genericity is free** — the abstraction compiles away (zero-cost abstraction), so you get reusability *without* performance cost. (5) **C++20 concepts** can constrain it: `template <std::floating_point T>` (or a custom numeric concept) documents the requirement and gives clear error messages. **Why NOT virtual functions:** a virtual approach (a base `Number` class with virtual arithmetic) would add a *runtime* virtual-call cost to *every arithmetic operation* — catastrophic for a hot-path numerical loop (you'd pay an un-inlinable indirect call per add/multiply), and it forces heap allocation and boxing of numbers. Virtual functions are for *runtime* polymorphism (unknown types, mixed collections), which you *don't* need here. **Why NOT (just) function overloading:** you *could* write separate overloads (`present_value(vector<double>, double)`, `present_value(vector<float>, float)`, ...) — but that's *manual code duplication*: you'd rewrite the *same logic* for each type, violating DRY, and adding a new type means copy-pasting the whole function again. Templates *automate* exactly this — one source, the compiler generates the overloads. (Overloading *is* right when the *logic differs* per type, but here the logic is *identical*, just the type varies — the template case.) **The trade-offs to acknowledge:** templates cause (a) *code bloat* (a copy per instantiated type — usually fine, and the linker can fold identical instantiations), (b) *longer compile times*, and (c) *historically cryptic error messages* (much improved by C++20 concepts). For a *numeric* library where types are compile-time-known and performance is critical, these costs are *well worth* the zero-runtime-overhead genericity. **The deep lesson:** this is the canonical "templates for compile-time genericity + performance" case — when the type is known at compile time and performance matters, templates give you *reusable, fast, write-once* code that virtual functions (runtime cost) and manual overloading (code duplication) can't match. Match the polymorphism mechanism to the requirement: *compile-time-known types + hot path → templates*; *runtime-decided types + flexibility → virtual functions*; *genuinely different logic per type → overloading*. The PV function is squarely templates — generic, zero-cost, and fast enough for the pricing hot path. This is *exactly* how real quant libraries (QuantLib, internal pricers) are built: heavily templated on the numeric/payoff/process types, achieving both broad reusability and production performance.)

**Career connection:** "what are templates, and when do you use them vs virtual functions?" Strong answer: **templates** are compile-time generic programming — you write code *once* parameterized by type, and the compiler **generates a specialized, fully-optimized version for each type at compile time** (static polymorphism), so it's **as fast as hand-written code with zero runtime overhead** (no boxing, no dispatch) — the foundation of the STL and of C++'s zero-cost abstraction. **Virtual functions** are *runtime* polymorphism — resolved at runtime via a vtable (an indirect call), giving *flexibility* (handle types unknown at compile time, mixed collections) at the cost of a *runtime* indirect call that *can't be inlined* (a few ns + possible cache miss/branch misprediction). **Choose templates when** the type is known at compile time and performance matters (the hot path) — zero overhead, inlinable; **choose virtual functions when** you need runtime flexibility (e.g. a container of different strategy objects decided at runtime) and the small virtual-call cost is acceptable. In *latency-critical* trading code I prefer templates to avoid virtual-call overhead; in *flexible architecture* code, virtual functions are cleaner. Bonus: modern features — **`constexpr`** (compile-time computation → zero runtime cost), **concepts** (C++20, constrain templates for clear errors), **variadic templates** — let you push even more work to compile time so the hot path stays lean. Naming static-vs-dynamic polymorphism, the inlining/vtable trade-off, and the hot-path reasoning is the senior C++ answer.

## Block 6 — C++ (systems) · A templated quant function & static vs dynamic
```cpp
#include <vector>
#include <iostream>
#include <cmath>
#include <memory>

// TEMPLATE: generic present-value, works for double/float/custom, zero overhead, inlinable
template <typename T>
T present_value(const std::vector<T>& cashflows, T rate) {
    T pv = 0;
    for (std::size_t t = 0; t < cashflows.size(); ++t)
        pv += cashflows[t] / std::pow(T(1) + rate, T(t + 1));
    return pv;                                   // compiler specializes per type T
}

// VIRTUAL: runtime polymorphism for *strategy selection* (flexibility, off hot path)
struct Strategy { virtual double signal(double px) const = 0; virtual ~Strategy() = default; };
struct Momentum  : Strategy { double signal(double px) const override { return px > 100 ? 1 : -1; } };
struct Reversion : Strategy { double signal(double px) const override { return px > 100 ? -1 : 1; } };

int main() {
    std::vector<double> cf_d = {100, 100, 100, 1100};
    std::vector<float>  cf_f = {100, 100, 100, 1100};
    std::cout << "PV (double): " << present_value(cf_d, 0.05) << "\n";   // double specialization
    std::cout << "PV (float):  " << present_value(cf_f, 0.05f) << "\n";  // float specialization

    // runtime polymorphism: choose strategy at runtime, store in one container
    std::vector<std::unique_ptr<Strategy>> strats;
    strats.push_back(std::make_unique<Momentum>());
    strats.push_back(std::make_unique<Reversion>());
    for (auto& s : strats) std::cout << "signal@105: " << s->signal(105) << "  ";
    std::cout << "\n";
}
```
The **template** `present_value` works for `double` *and* `float` (compiler generates each, zero overhead), while **virtual** `Strategy` gives runtime flexibility to hold and dispatch different strategies — each polymorphism mechanism used where it fits.

**Linux:** `g++ -O3 -std=c++20 templates.cpp -o templates && ./templates` — `-std=c++20` enables concepts; inspect the optimized assembly with `g++ -O3 -S` to *see* the template specializations compile to type-specific instructions.

## Quant Thinking Track — The right abstraction, resolved at the right time
Templates teach that you can have genericity, reusability, *and* performance — if the abstraction is resolved at *compile time*. The deeper habit: ask *when* computation happens (compile time vs runtime) and push as much as possible to compile time, so the runtime hot path stays lean. A good generic abstraction captures shared structure once and specializes optimally — the code-level echo of finding the unifying framework (state-space, factor models) that contains the special cases. Abstract *and* fast is achievable; the key is resolving the abstraction before runtime.
