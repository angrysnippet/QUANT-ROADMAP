## Block 1 — What ML actually is
A model is a function `f(features) → prediction` whose parameters are **learned from data** by minimising a loss. The goal is **generalization** — low error on data it has never seen.

- **Supervised:** learn from labelled examples (features → target). *Regression* (numeric target, e.g. next-day return) and *classification* (categorical, e.g. up/down).
- **Unsupervised:** find structure without labels (clustering, PCA — Day in the linear-algebra module).

*Why it matters:* most quant ML is supervised prediction of returns/risk from features — and it lives or dies by generalization, not in-sample fit.

---

## Block 2 — Features, labels, and the pipeline
```
raw data → features (X) + labels (y) → train model → validate → test → deploy → monitor
```
**Feature engineering** (turning prices/fundamentals into predictive inputs) usually matters more than the model choice. Garbage features → garbage model, however fancy.

**Concept check:** for "predict tomorrow's return", what's a label and what's a feature? (Label = next-day return; features = today-and-earlier signals only — never future info.)

## Block 3 — Generalization & validation (recap, applied)
ML formalises Day 127: train/validation/test, cross-validation, regularization. The **only** score that matters is out-of-sample. Everything from the stats block (overfitting, bias-variance, multiple testing, power) **is** the ML discipline.

---

## Block 4 — Finance-specific ML pitfalls (this is where people fail)
1. **Lookahead/leakage** — features must use only past info (Day 124).
2. **Non-stationarity** — the data-generating process drifts; a model trained on 2010 may be wrong in 2020 (Day 142).
3. **Autocorrelation** — i.i.d. assumptions break; use **purged, embargoed, walk-forward CV**, not random k-fold.
4. **Low signal-to-noise** — favour simple, regularized models (Day 128).
5. **Regime changes & overfitting to crises.**

**Exercise:** why is plain random k-fold cross-validation *dangerous* on time-series returns? (It trains on future data relative to some validation points → leakage → inflated scores.)

---

## Block 5 — Quant Thinking (ML is a tool, not magic)

- ML doesn't create signal — it **extracts** signal that exists. In near-efficient markets, that signal is tiny, so ML's edge over simple models is often small and fragile.
- The winning quant ML workflow is **discipline-heavy**: careful labels, leakage-proof features, walk-forward validation, regularization, and modest expectations.
- **Interpretability matters** for risk: a model you can't explain is one you can't trust with capital when it behaves oddly.

**Hard puzzle:** a deep model predicts returns with 95% in-sample accuracy. Your reaction? (Alarm — near-certainly leakage or overfit; markets are not 95%-predictable. Hunt for the bug before celebrating.)

**Career connection:** firms probe whether you understand *ML's limits in finance* (leakage, non-stationarity, CV pitfalls) far more than whether you can call `model.fit()`. Judgment > library knowledge.

## Block 6 — Python (quant) · scikit-learn workflow skeleton
```python
import numpy as np
from sklearn.linear_model import Ridge
from sklearn.model_selection import TimeSeriesSplit  # NOT KFold for time series
rng = np.random.default_rng(0)
X = rng.normal(size=(500, 3)); y = X @ [0.3,-0.1,0.0] + rng.normal(0,1,500)  # mostly noise
tscv = TimeSeriesSplit(n_splits=5)
for tr, te in tscv.split(X):
    m = Ridge(alpha=1.0).fit(X[tr], y[tr])
    print(round(m.score(X[te], y[te]), 3))   # modest/again small OOS R² — realistic
```
