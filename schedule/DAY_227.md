## Block 1 — Floating point: not the reals
Computers represent real numbers as **floating point** (IEEE 754 `double`/`float`) — a *finite* approximation that behaves *subtly differently* from real-number math, and ignoring this causes bugs:
- **not all numbers are representable** — `0.1` has *no exact* binary representation (like ⅓ in decimal), so `0.1 + 0.2 != 0.3` *exactly* (it's `0.30000000000000004`). **Never compare floats with `==`** — use a tolerance (`abs(a-b) < epsilon`),
- **rounding error accumulates** — each operation rounds; summing millions of numbers accumulates error,
- **catastrophic cancellation** — subtracting two nearly-equal large numbers *loses precision* dramatically (the significant digits cancel, leaving amplified rounding noise) — a major source of numerical bugs in finance (e.g. computing a small difference of large present values),
- **limited precision** — `double` has ~15-16 significant decimal digits, `float` ~7; beyond that, precision is *gone*,
- **special values** — `inf`, `-inf`, `NaN` (Not-a-Number, e.g. `0.0/0.0`); `NaN` *poisons* computations (`NaN` compared to anything is false, propagates through arithmetic) — a `NaN` in a price feed can silently corrupt everything downstream.

**This is not pedantry — floating-point errors have caused real financial losses and even disasters** (a famous one: the Patriot missile failure from accumulated float error; in finance, mispriced trades from precision loss). The quant developer *must* understand floating point.

*Why it matters:* you're doing *financial* math where errors mean *money*. A precision bug can mis-price a trade, miscompute risk, or silently corrupt a backtest — and these bugs are *subtle* (the code "looks right").

---

## Block 2 — Numerical stability
Beyond representation, *how you compute* affects accuracy — **numerical stability** is choosing algorithms that *don't amplify* error:
- **avoid catastrophic cancellation** — reformulate to avoid subtracting nearly-equal numbers (e.g. the numerically-stable quadratic formula, or computing variance with a stable one-pass algorithm — Welford's — rather than the naive `E[X²]−E[X]²` which catastrophically cancels for large means),
- **summation order matters** — summing many floats, adding *small to large* (or using **Kahan summation**, which tracks the lost low-order bits) is *far* more accurate than naive summation; for a million-element price sum, naive summation can lose meaningful precision,
- **conditioning** — some problems are *inherently* sensitive (ill-conditioned — e.g. inverting a near-singular matrix, Day 211's noisy Σ!); a *stable algorithm* on an *ill-conditioned problem* still struggles, so you need *both* good algorithms *and* well-conditioned problems (factor models condition Σ, Day 211),
- **use appropriate precision** — `double` for most finance (the default); `float` only when speed/memory justify it *and* precision allows; sometimes *higher* precision or *integer/fixed-point* (e.g. representing prices in integer cents/ticks to avoid float entirely — common in exchanges and accounting where *exactness* matters).

The discipline: **understand where your computation loses precision, and reformulate for stability** — especially in finance where the numbers represent money and errors compound.

## Block 3 — SIMD: one instruction, many numbers
**SIMD (Single Instruction, Multiple Data)** is hardware parallelism *within* a core: a single instruction operates on *multiple* numbers *simultaneously*. Modern CPUs have SIMD registers (SSE: 4 floats, AVX2: 8 floats, AVX-512: 16 floats) and instructions that, e.g., *add 8 pairs of floats in one instruction*:
- **vectorization** — processing arrays with SIMD gives a *4–16× speedup* on numerical loops (the SIMD width) — *enormous* for the array-heavy math of quant systems (pricing portfolios, computing signals over many assets, Monte Carlo, matrix math),
- **how to get it:**
  - **auto-vectorization** — the compiler *automatically* vectorizes simple loops *if* the data is laid out well (contiguous — Day 222's SoA!) and the loop is simple (no complex dependencies/branches). Compile with `-O3 -march=native` (Day 221) to enable it,
  - **compiler hints / restrict** — help the compiler vectorize (e.g. `__restrict` to promise no aliasing),
  - **intrinsics** — *manually* write SIMD using compiler intrinsics (`_mm256_add_pd` etc.) for full control when auto-vectorization isn't enough — powerful but verbose and architecture-specific,
  - **libraries** — Eigen, xsimd, std::simd (C++26) provide higher-level vectorized math.

The connection to Day 222: **SIMD needs cache-friendly, contiguous data (SoA)** — you *can't* vectorize strided/pointer-chasing access. So data-oriented design (Day 222) and SIMD go hand-in-hand: lay out data contiguously (SoA), and the compiler vectorizes it for a *massive* speedup. This is *why* SoA + SIMD is the foundation of high-performance numerical code.

**Concept check:** you have a loop computing `result[i] = a[i] * b[i] + c[i]` over a million contiguous `double` arrays. How can it run ~4-8× faster with no algorithm change, and what enables it? (**SIMD vectorization** — the loop does the *same* operation (multiply-add) on *independent* elements, so the CPU can process *multiple* elements *per instruction* using SIMD registers (AVX2: 4 doubles at once; AVX-512: 8). With `-O3 -march=native`, the compiler *auto-vectorizes* this simple loop — generating SIMD instructions that compute 4-8 results per iteration instead of 1, a ~4-8× speedup *with no code change* (it's even better with FMA — fused multiply-add — doing the `*` and `+` in one instruction). **What enables it:** (1) **contiguous data** (the arrays are `std::vector<double>`, contiguous — Day 222) so SIMD can load 4-8 consecutive elements at once; (2) **independent iterations** (no data dependency between elements — each `result[i]` is independent) so they can be computed in parallel; (3) **simple loop body** (no complex branches/calls that block vectorization); (4) **compiler optimization enabled** (`-O3 -march=native` to use the CPU's SIMD width). The lesson: *lay out data contiguously (SoA, Day 222), write simple loops, compile with optimization, and the compiler gives you a 4-16× speedup for free via SIMD* — which is *why* data-oriented design and SIMD are the one-two punch of high-performance numerics. If the data were strided (AoS) or pointer-chasing, SIMD *couldn't* apply — cache-friendly layout is the *prerequisite* for vectorization.)

## Block 4 — Mathematics · precision, performance, and their tension
The numerical-computing core is a *tension* between *speed* and *accuracy*, and the mathematics of managing it:
- **the precision/performance trade-off** — `float` (32-bit) is *half the memory* of `double` (64-bit), so it's *faster* (more fit in cache and SIMD registers — *2× the SIMD width*: AVX2 does 8 floats vs 4 doubles) — but has *less precision* (~7 vs ~16 digits). So you *can* use `float` for a 2× speedup *if* the precision suffices (e.g. some ML inference, some signals) — but for *financial* calculations where precision matters (accumulating P&L, pricing), `double` is usually mandatory, and `float`'s precision loss can cause real errors. The choice is a *deliberate* trade-off: profile the precision needs *and* the performance needs,
- **error analysis** — *forward error* (how wrong is the answer?) vs *backward error* (the answer is exact for *slightly perturbed* inputs); a *stable* algorithm has small backward error. *Condition number* measures how much input error amplifies into output error (an ill-conditioned problem — like inverting noisy Σ, Day 211 — amplifies error regardless of algorithm). The practical synthesis: **accuracy = condition number × algorithm stability × precision** — you need a well-conditioned problem, a stable algorithm, *and* adequate precision; weakness in any one loses accuracy,
- **when exactness matters, avoid floats entirely** — for *money* and *counts* where *exactness* is required (accounting, order sizes, prices in ticks), use **integers** (cents, ticks, basis points as integers) or **fixed-point** arithmetic — *exact*, no rounding, no `0.1+0.2` surprises. Exchanges and accounting systems often represent prices as *integer ticks* precisely to avoid floating-point error in money. The lesson: *floats are for approximate continuous math (pricing, risk, signals); integers/fixed-point are for exact money/counts.*

The unifying mathematical insight: **numerical computing is the disciplined management of finite-precision arithmetic — understanding where precision is lost (representation, cancellation, accumulation, conditioning), choosing stable algorithms and appropriate precision, and balancing the speed/accuracy trade-off — all while exploiting SIMD parallelism (enabled by cache-friendly layout) for performance.** It's where *mathematical correctness* meets *hardware performance*, and a quant developer must master *both* — fast math that's *wrong* is worse than useless (it loses money silently), and correct math that's *too slow* can't be deployed.

## Block 5 — Quant Thinking · fast AND correct — and respect finite precision
Numerical computing crystallizes a principle essential to quant systems: **you must be *both* fast *and* correct — and in finance, where the numbers are money, the "correct" part demands a deep respect for finite-precision arithmetic that researchers and naive programmers routinely underestimate.** This has two equally-important dimensions. **First, correctness — respect floating point.** It is *dangerously* easy to treat computer floating-point numbers as if they were real numbers, but they're a *finite approximation* with subtle, treacherous behavior: `0.1+0.2 != 0.3`, catastrophic cancellation, accumulating rounding error, `NaN` poisoning, limited precision. In *most* domains these errors are negligible; in *finance*, where the numbers represent *money* and errors *compound* (over millions of trades, accumulating P&L, risk calculations), floating-point bugs can cause *real, material financial losses* — mispriced trades, miscalculated risk, silently corrupted backtests — and they're *insidious* because the code *looks* correct (no crash, no error, just a *wrong number*). So the quant developer must understand numerical analysis: where precision is lost (representation, cancellation, summation order, conditioning), how to write *numerically stable* algorithms (Kahan summation, stable variance, avoiding cancellation), when to use higher precision or *exact* integer/fixed-point arithmetic (for money/counts), and *never* to compare floats with `==`. This connects to the curriculum's spine of *intellectual honesty and rigor*: just as you must be honest about statistical uncertainty (don't trust a backtest, Day 199) and model uncertainty (robust optimization, Day 218), you must be honest about *numerical* uncertainty — your computations have *finite precision*, and pretending otherwise causes bugs that cost money. **Second, performance — exploit the hardware (SIMD).** Quant systems do *enormous* amounts of numerical work (pricing portfolios, risk over thousands of scenarios, signals over many assets, Monte Carlo with millions of paths, matrix operations), and **SIMD vectorization** offers a *4-16× speedup* on this array-heavy math *for free* — *if* the data is laid out cache-friendly (SoA, contiguous — Day 222) and the loops are simple. So *data-oriented design (Day 222) and SIMD are the foundation of fast numerics*: contiguous data → auto-vectorization → massive speedup. This *directly* enables research velocity (faster backtests → more ideas tested → more alpha found) and production performance (faster pricing/risk → react faster, handle more). **The synthesis — fast AND correct:** the quant developer must deliver *both* — numerical code that is *correct* (numerically stable, precision-aware, respecting floating point and money) *and* fast (SIMD-vectorized, cache-friendly) — because fast-but-wrong math *silently loses money* (the worst kind of bug — undetected) and correct-but-slow math *can't be deployed* (misses the latency/throughput requirements). This is the engineering culmination where *mathematical rigor* (the numerical-analysis correctness of the nine pillars' formulas) meets *systems performance* (the hardware-exploiting speed of this module): a pricing formula (Black-Scholes, Day 153) is only *useful* if it's implemented *both* correctly (stable, precise — a precision bug mis-prices the option) *and* fast (vectorized — to price a whole portfolio in microseconds). The unifying wisdom: **master numerical computing as the discipline of being *both* fast *and* correct with finite-precision arithmetic — respect floating point's subtleties (correctness, especially with money, where errors compound and hide), write numerically stable algorithms, use appropriate precision (or exact integers for money), *and* exploit SIMD vectorization (enabled by cache-friendly data layout) for the speed quant systems demand.** The researcher who treats numbers as abstract reals and the programmer who only chases speed both fail; the quant developer who is *rigorous about precision* (correct) *and* *exploits the hardware* (fast) builds the numerical systems — pricers, risk engines, backtesters, signal computations — that are the computational heart of quant finance, turning the *mathematics* of the theory into *correct, fast, deployable* code. In a field where numbers are money, being *both* fast *and* numerically correct isn't optional — it's the job.

**Hard puzzle:** a risk system computes portfolio variance using the textbook formula `Var = E[X²] − (E[X])²` over a large set of P&L values that have a large mean (the portfolio has been profitable, so values are large positive numbers). The computed variance is occasionally *negative* (which is mathematically impossible) or wildly wrong. Diagnose the numerical problem and fix it. (**Diagnosis — catastrophic cancellation in the textbook variance formula:** the formula `Var = E[X²] − (E[X])²` is *mathematically* correct but *numerically unstable* when the mean is large relative to the variance — exactly this case (large positive P&L values, so large mean). Here's why: if the values have a large mean `μ` (say ~1,000,000) but small variance (say ~100), then `E[X²]` ≈ `μ² + Var` ≈ `1,000,000,000,100` and `(E[X])²` ≈ `1,000,000,000,000` — two *huge*, *nearly-equal* numbers. Subtracting them (`E[X²] − (E[X])²`) is **catastrophic cancellation** (Block 1): the large common part (`μ² ≈ 10¹²`) cancels, and what's left (the variance ~100) is computed from the *difference of two numbers that agree in their first ~12 digits* — but `double` only has ~15-16 significant digits, so after the cancellation, only ~3-4 digits of the variance survive, *amplified by rounding noise*. The result is *wildly inaccurate*, and rounding error can even make the (tiny, noise-dominated) difference *negative* — giving an "impossible" negative variance. The textbook formula *catastrophically loses precision* precisely when the mean dominates the variance. **The fix — use a numerically stable algorithm:** (1) **Two-pass algorithm** — first compute the mean `μ` (one pass), then compute `Var = mean of (X − μ)²` (second pass). By subtracting the mean *first*, the `(X − μ)` values are *small* (no large common part), so squaring and averaging them has *no cancellation* — numerically stable and accurate. (2) **Welford's online algorithm** (one-pass, numerically stable) — incrementally updates the mean and variance in a single pass *without* the cancellation, ideal for streaming data (you don't need to store all values) — the standard choice for a streaming risk system. (3) For *parallel/SIMD* computation, use a numerically-stable parallel variant (Chan's parallel algorithm that combines partial results stably). **The deep lesson:** the bug isn't in the *math* (the formula is algebraically correct) — it's in the *numerical implementation*: `E[X²]−(E[X])²` is **numerically unstable** due to catastrophic cancellation when the mean is large, a *classic*, *well-known* numerical-analysis pitfall that *looks* correct and passes casual testing (it's fine when the mean is small) but *fails* — silently and sometimes catastrophically (negative variance!) — in production with large-mean data. The fix is a *numerically stable algorithm* (two-pass or Welford's) that avoids the cancellation. This is *exactly* the "fast but wrong silently loses money" danger (Block 5): a risk system computing *wrong* (or impossible negative) variance would mis-measure risk — potentially *catastrophically* (underestimating risk → over-leverage → blowup, Days 178/213). And it's *insidious* — no crash, no error, just a *wrong risk number* that looks plausible until it's impossibly negative. The quant developer *must* know numerical stability (catastrophic cancellation, stable variance algorithms) precisely because *financial* computations on *real money* demand correctness, and naive textbook formulas can fail silently. **Respect finite precision: use numerically stable algorithms, especially for financial calculations where the numbers are money and a precision bug mis-measures risk or mis-prices trades.** This single example — the unstable vs stable variance computation — is a canonical lesson that *mathematical correctness ≠ numerical correctness*, and that the quant developer needs *both*.)

**Career connection:** "what numerical issues matter in quant systems, and how do you write fast numerical code?" Strong answer — *two halves, correctness and speed*: **Correctness (respect floating point):** computers use finite-precision **floating point** that behaves subtly differently from real numbers — `0.1+0.2 != 0.3` (never compare floats with `==`), **catastrophic cancellation** (subtracting nearly-equal large numbers destroys precision — e.g. the textbook variance `E[X²]−(E[X])²` fails for large-mean data, even giving negative variance), accumulating rounding error, and `NaN` poisoning. In *finance*, where numbers are *money* and errors *compound*, these cause *real, silent* losses (mispriced trades, miscalculated risk), so I use **numerically stable algorithms** (two-pass/Welford's variance, Kahan summation, stable reformulations), appropriate **precision** (`double` for finance; **integer/fixed-point** for *exact* money/counts like prices-in-ticks), and respect conditioning (a well-conditioned problem matters too — Day 211's Σ). **Speed (exploit SIMD):** quant systems do massive array math, and **SIMD vectorization** gives a **4-16× speedup** (one instruction processes 4-16 numbers) — *for free* via compiler **auto-vectorization** (`-O3 -march=native`) *if* the data is **cache-friendly/contiguous (SoA, Day 222)** and loops are simple; for full control, **intrinsics** or libraries (Eigen, std::simd). Data-oriented design (contiguous SoA) and SIMD are the one-two punch of fast numerics. **The synthesis:** deliver *both* — *correct* (numerically stable, precision-aware, money-respecting) *and* *fast* (SIMD-vectorized, cache-friendly) — because fast-but-wrong *silently loses money* and correct-but-slow *can't deploy*. Naming floating-point pitfalls (cancellation, `==`, `NaN`), numerical stability (stable variance/Kahan), precision choices (double/int/fixed-point for money), and SIMD/auto-vectorization (enabled by SoA) — framed as *both fast and correct* — is the senior numerical-quant-developer answer.

## Block 6 — C++ (systems) · SIMD speedup & stable vs unstable variance
```cpp
#include <vector>
#include <cmath>
#include <chrono>
#include <iostream>

// auto-vectorized (SIMD) multiply-add over contiguous arrays — compiler vectorizes with -O3 -march=native
void fma_loop(const std::vector<double>& a, const std::vector<double>& b,
              const std::vector<double>& c, std::vector<double>& out) {
    for (std::size_t i = 0; i < a.size(); ++i)
        out[i] = a[i] * b[i] + c[i];                 // simple, contiguous → SIMD-vectorizable
}

// UNSTABLE vs STABLE variance — the catastrophic-cancellation lesson
double var_unstable(const std::vector<double>& x) {  // E[X²] − (E[X])²  : cancels for large mean
    double s = 0, s2 = 0;
    for (double v : x) { s += v; s2 += v*v; }
    double mean = s / x.size();
    return s2/x.size() - mean*mean;                  // catastrophic cancellation!
}
double var_stable(const std::vector<double>& x) {    // two-pass: subtract mean first
    double mean = 0; for (double v : x) mean += v; mean /= x.size();
    double s = 0; for (double v : x) { double d = v - mean; s += d*d; }
    return s / x.size();                             // stable — no cancellation
}

int main() {
    int N = 1'000'000;
    std::vector<double> a(N,1.5), b(N,2.0), c(N,0.5), out(N);
    auto t0 = std::chrono::high_resolution_clock::now();
    for (int r=0;r<100;++r) fma_loop(a,b,c,out);
    auto t1 = std::chrono::high_resolution_clock::now();
    std::cout << "SIMD fma loop: "
              << std::chrono::duration_cast<std::chrono::microseconds>(t1-t0).count() << " us (100 reps)\n";

    // values with a LARGE mean and small variance — where the unstable formula fails
    std::vector<double> pnl(N);
    for (int i=0;i<N;++i) pnl[i] = 1'000'000.0 + (i % 100) * 0.01;   // big mean, tiny variance
    std::cout << "unstable variance: " << var_unstable(pnl) << "  <- WRONG (cancellation, may be <0)\n";
    std::cout << "stable variance:   " << var_stable(pnl)   << "  <- correct\n";
}
```
The simple contiguous loop **auto-vectorizes** to SIMD (4-8× faster with `-march=native`), and the **stable** two-pass variance gives the correct answer where the textbook **unstable** formula fails catastrophically on large-mean data — fast *and* correct, the dual mandate of quant numerics.

**Linux:** `g++ -O3 -march=native simd.cpp -o simd && ./simd`; verify vectorization with `g++ -O3 -march=native -fopt-info-vec simd.cpp` (reports which loops vectorized), and inspect the SIMD assembly with `-S`.
