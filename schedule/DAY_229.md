## Block 1 — Testing: verifying correctness
You can't *trust* code you haven't *tested* — and in finance, untested code that handles money is a liability. The testing toolkit:
- **unit tests** — test individual functions/components in isolation (does `present_value` compute correctly? does the SMA handle edge cases?). Fast, focused, the foundation. Use a framework (Google Test, Catch2),
- **integration tests** — test components *working together* (does the strategy → risk → execution flow work end-to-end?),
- **regression tests** — assert that behavior *doesn't change* unexpectedly (the research-production parity test of Day 228 is a regression test; also: "this strategy on this data produces *exactly* these results" — catches accidental changes),
- **property-based testing** — instead of specific cases, assert *properties* hold for *many random inputs* (e.g. "portfolio variance is always ≥ 0" — would have caught the Day 227 unstable-variance bug!; "a round-trip serialize/deserialize is identity"),
- **edge cases** — *especially* important in finance: empty data, single element, market open/close, zero/negative prices, `NaN`s (Day 227), extreme values, the exact boundary of a risk limit. Bugs *hide* in edge cases.

The discipline: **test thoroughly, especially edge cases and money-handling logic, because a bug in production trades real capital.** Tests also *enable change* — you can refactor/optimize confidently because tests catch regressions. **Untested code is broken code you haven't found yet.**

*Why it matters:* a correctness bug in a trading system can lose real money (a mispriced trade, a risk limit not enforced, a double-sent order). Testing is how you *prevent* catastrophe, not just polish.

## Block 2 — Profiling: finding the *real* bottleneck
The cardinal rule of optimization (Day 222): **profile, don't guess** — *measure* where time is actually spent, because performance intuition is *notoriously* unreliable (the bottleneck is rarely where you think). The profiling toolkit:
- **CPU profilers** (`perf`, VTune, gprof) — show *where* the program spends time (which functions, which lines) — so you optimize the *actual* hot spots, not guessed ones,
- **the 80/20 rule (Amdahl's Law)** — usually a *small* fraction of the code accounts for *most* of the runtime; profiling *finds* that fraction so you optimize *it* (and leave the rest readable). *Optimizing non-bottleneck code is wasted effort* (and harms maintainability),
- **hardware counters** (`perf stat`) — cache misses (Day 222), branch mispredictions (Day 226), instructions-per-cycle — reveal *why* code is slow (memory-bound? branch-bound?), guiding the *right* optimization,
- **micro-benchmarking** — measure specific functions in isolation (carefully — avoid the compiler optimizing away your benchmark, use realistic data, measure the *distribution* including the tail — Day 226),
- **measure before *and* after** — confirm your optimization *actually* helped (sometimes it doesn't, or makes it worse — *measure* to know).

The discipline: **profile to find the real bottleneck (usually a small fraction of the code), optimize *that*, measure to confirm — never optimize blindly by intuition.** This is *empirical* performance engineering: the *profiler*, not your guess, is the authority (the same measure-don't-guess intellectual honesty as Day 222).

## Block 3 — Debugging and sanitizers
When code is *broken* (wrong output, crash, leak, race), you must *find and fix* it — and the right tools turn hours of pain into minutes:
- **debuggers** (`gdb`, IDE debuggers) — step through code, inspect state, set breakpoints/watchpoints, examine the call stack at a crash — the fundamental tool for *understanding* what the code actually does (vs what you think it does),
- **sanitizers** (the modern superpower) — compiler tools that *instrument* the code to *catch* whole bug classes at runtime:
  - **AddressSanitizer (ASan)** — catches memory errors: buffer overflows, use-after-free, leaks (`-fsanitize=address`),
  - **ThreadSanitizer (TSan)** — catches **data races** (Day 225) — *essential* for concurrent code, since races are otherwise nearly impossible to find (`-fsanitize=thread`),
  - **UndefinedBehaviorSanitizer (UBSan)** — catches undefined behavior (integer overflow, bad casts, etc.),
- **valgrind** — detects memory leaks and errors (slower than ASan but thorough),
- **logging** — record key events/state (Day 228's logging component) so you can *reconstruct* what happened in production (where you can't attach a debugger) — *essential* for debugging *live* trading issues,
- **reproducibility** (Day 228 determinism) — a *deterministic* system lets you *reproduce* any bug (re-run with the same inputs/seed), which is *half* of fixing it; non-deterministic bugs (races, timing) are *far* harder, which is *why* determinism and TSan matter.

The discipline: **use the right tools — sanitizers catch entire bug classes automatically (run your tests under ASan/TSan/UBSan!), debuggers and logging let you understand failures, and determinism makes bugs reproducible.** Modern tooling makes whole categories of bugs *findable* that were once nearly impossible.

**Concept check:** you have a concurrent trading system that *occasionally* (once a day) produces corrupted order data, but you *can't reproduce* it on demand and normal testing passes. What tool would you reach for, and why? (**ThreadSanitizer (TSan)** — `-fsanitize=thread`. The symptoms — *occasional*, *non-reproducible*, corruption in *concurrent* code, passing normal tests — are the *classic signature* of a **data race** (Day 225): two threads accessing shared data without proper synchronization, where the corruption only manifests under *specific timing* (hence rare and non-reproducible). Normal testing *can't* reliably trigger it (it depends on thread timing). **TSan instruments the code to *detect* data races** — it tracks memory accesses across threads and *flags* when two threads access the same memory concurrently without synchronization, *even if the race didn't corrupt anything on that run*. So TSan finds the race *deterministically* (it detects the *unsynchronized access pattern*, not just the rare *corruption*), pointing you *exactly* to the unsynchronized memory and the two threads involved — turning a maddening once-a-day heisenbug into a precise, fixable finding. This is *why* TSan is essential for concurrent code: data races are otherwise *nearly impossible* to find by normal testing/debugging (they're non-deterministic and timing-dependent), but TSan makes them *findable*. Run *all* concurrent code under TSan as part of testing. The lesson: *match the tool to the bug class — sanitizers (especially TSan for races) catch entire categories of bugs that defeat normal testing*, and concurrent corruption that's rare and non-reproducible screams "data race → TSan.")

## Block 4 — Mathematics · the economics of bugs and the testing/performance disciplines
The deeper framework for *why* and *how much* to test, profile, and debug:
- **the cost of bugs grows exponentially with how late they're caught** — a bug caught in *development* (by a test) costs minutes; caught in *code review* costs hours; caught in *production* costs *real money* and *reputation* (and in trading, a production bug can lose *millions* in seconds — a mispriced trade, an unchecked risk limit, a runaway order loop). So the *economics* strongly favor *catching bugs early* — which is *why* testing (catch in development), code review, sanitizers (catch automatically), and staged rollout (Day 228) are *investments* that pay for themselves *many* times over. *The cheapest bug is the one caught by a test before it ships*,
- **the testing trade-off** — testing has a *cost* (time to write/maintain tests) and a *benefit* (bugs caught early, confidence to change), and the *optimal* amount of testing depends on the *stakes*: for *money-handling, hard-to-change, production* code (a trading system), test *heavily* (the cost of a bug is enormous); for *throwaway research scripts*, test *lightly* (run-once, low stakes — though *validation* discipline still applies, Day 199). Match the testing rigor to the stakes,
- **profiling and Amdahl's Law** — Amdahl's Law quantifies the *limit* of optimization: if a part takes fraction `p` of runtime and you speed it up by factor `s`, the *overall* speedup is `1 / ((1-p) + p/s)`. The implication: **optimizing a part that's only 5% of runtime can *at most* give a ~5% overall speedup** — so *profile to find the part that's 80% of runtime* and optimize *that* for a big win. *This is the mathematical justification for "profile, don't guess"* — your optimization effort's *payoff* is bounded by the fraction of time the optimized part consumes, so you *must* find (profile) the *dominant* cost to optimize effectively. Optimizing non-bottlenecks is *mathematically* near-useless (Amdahl caps the gain),
- **the reliability/performance balance** — you need *both* correctness (testing/debugging) *and* speed (profiling/optimization), and they can *trade off* (optimized code is often more complex/fragile — Day 226's "optimize judiciously"). The resolution: *correctness first* (a fast wrong answer is worthless — Day 227), *then* optimize the *profiled* bottleneck *with tests guarding correctness* (so optimization doesn't break things). Tests *enable* safe optimization (you can refactor/optimize confidently because tests catch regressions).

The unifying insight: **testing, profiling, and debugging are the *empirical, disciplined* crafts that turn code into *reliable, fast* software — testing catches bugs early (where they're cheap), profiling finds the bottleneck that's worth optimizing (Amdahl), debugging/sanitizers find and fix what's broken, and the whole discipline is *economically* justified because in money-handling systems, the cost of an *uncaught* bug or an *un-optimized* bottleneck is enormous.** It's *intellectual honesty* applied to code: *measure* correctness (test) and performance (profile) rather than *assuming* them.

## Block 5 — Quant Thinking · reliable software is a discipline, not an afterthought
Testing, profiling, and debugging crystallize the final, essential principle of the systems module: **writing code is the easy part; making it *correct* and *fast* — and *knowing* (through measurement) that it is — is the hard part and the real job, achieved through the disciplines of testing, profiling, and debugging rather than through hope or cleverness.** This is the *professional* maturity that separates someone who *writes code* from someone who *ships reliable systems*. **First, correctness is not assumed — it's verified.** Untested code is *broken code you haven't found yet* — and in a system handling *real money*, an untested bug is a *financial catastrophe waiting to happen* (a mispriced trade, an unenforced risk limit, a runaway order). So the professional *tests thoroughly* — unit, integration, regression, property-based, *especially edge cases and money logic* — and runs tests *under sanitizers* (ASan/TSan/UBSan) to catch whole bug classes *automatically*. Testing isn't *polish* applied at the end; it's a *discipline* woven throughout, *enabling* confident change and *preventing* catastrophe. This is the same *intellectual honesty* as the research discipline (don't trust a backtest, Day 199; don't trust intuition, measure): *don't trust that your code is correct — verify it with tests.* **Second, performance is not guessed — it's measured.** The cardinal rule (Day 222): *profile, don't guess* — performance intuition is unreliable, and Amdahl's Law (Block 4) means optimizing a non-bottleneck is *mathematically* near-useless. So the professional *profiles* to find the *actual* bottleneck (usually a small fraction of the code), optimizes *that* (with tests guarding correctness), and *measures* to confirm the improvement — *empirical* performance engineering, not blind optimization. This *also* respects maintainability: you optimize *only* the profiled hot path, keeping the rest *simple and clear* (premature/scattered optimization creates fragile, unmaintainable systems — Day 226). **Third, debugging is a *craft* with *tools*** — when code breaks (and it will), the professional uses the *right tools* (debuggers, sanitizers, logging, determinism for reproducibility) to *understand and fix* the failure *systematically*, rather than flailing with print statements. Modern sanitizers especially turn once-impossible bugs (data races, memory corruption) into *findable* ones — a superpower the professional *uses* (runs all concurrent code under TSan, all code under ASan). **Fourth, the economics justify the rigor** — the cost of a bug grows *exponentially* with how late it's caught (minutes in development → millions in production), so testing/profiling/debugging are *investments* that pay for themselves *many* times over in a high-stakes, money-handling system. *Skipping* them isn't "moving fast"; it's *deferring* a *larger* cost (a production catastrophe) — false economy. This connects to the entire curriculum's spine of *rigor, measurement, and intellectual honesty*: just as honest *research* requires verifying (not assuming) that an edge is real (Days 174/199), honest *engineering* requires verifying (not assuming) that code is correct (testing) and fast (profiling) — *measuring* rather than *hoping*. And it connects to *structural correctness* (Day 224): testing *complements* structural correctness (encode invariants to *prevent* bug classes, *and* test to *catch* the rest). The unifying wisdom: **reliable, fast software is achieved through the *disciplines* of testing (verify correctness, especially edge cases and money logic, run under sanitizers), profiling (measure the real bottleneck — Amdahl — and optimize *that* with tests guarding correctness), and debugging (use the right tools — debuggers, sanitizers, logging, determinism — to find and fix systematically) — not through hope, cleverness, or assumption — because in money-handling systems, the cost of an uncaught bug or unmeasured bottleneck is enormous, and the only way to *know* your code is correct and fast is to *measure* it.** The quant developer who masters these crafts ships systems that are *trustworthy* (tested, debugged — won't lose money to bugs) and *fast* (profiled, optimized where it matters) — turning the *theory* (the nine pillars) and the *architecture* (Day 228) into a *reliable, deployed, money-making* reality. This is the engineering professionalism that the best firms demand: not just *writing* the trading system, but *verifying* — through disciplined testing, profiling, and debugging — that it's *correct and fast enough to trust with real capital*. **Code is a hypothesis; testing, profiling, and debugging are how you verify it — and in finance, verification isn't optional, because the cost of being wrong is real money.**

## Block 6 — C++ (systems) · A unit test, a benchmark, and sanitizer usage
```cpp
#include <vector>
#include <cassert>
#include <cmath>
#include <chrono>
#include <iostream>

double stable_variance(const std::vector<double>& x) {           // (Day 227)
    if (x.empty()) return 0.0;                                    // edge case!
    double mean = 0; for (double v : x) mean += v; mean /= x.size();
    double s = 0; for (double v : x) { double d = v - mean; s += d*d; }
    return s / x.size();
}

// UNIT TESTS — verify correctness, including EDGE CASES and a PROPERTY
void test_variance() {
    assert(std::abs(stable_variance({2,4,4,4,5,5,7,9}) - 4.0) < 1e-9);  // known answer
    assert(stable_variance({}) == 0.0);                           // edge: empty
    assert(stable_variance({5.0}) == 0.0);                        // edge: single element
    // PROPERTY: variance is ALWAYS >= 0, even for large-mean data (the Day 227 bug!)
    std::vector<double> big(1000, 1e9); for (int i=0;i<1000;++i) big[i] += i*0.01;
    assert(stable_variance(big) >= 0.0);                          // would catch the unstable bug
    std::cout << "all variance tests passed\n";
}

int main() {
    test_variance();

    // MICRO-BENCHMARK — measure performance (profile-don't-guess in miniature)
    std::vector<double> data(1'000'000, 3.14);
    auto t0 = std::chrono::high_resolution_clock::now();
    volatile double v = 0;
    for (int r = 0; r < 100; ++r) v = stable_variance(data);
    auto t1 = std::chrono::high_resolution_clock::now();
    std::cout << "stable_variance: "
              << std::chrono::duration_cast<std::chrono::microseconds>(t1-t0).count()
              << " us (100 reps) — profile to confirm this is/isn't a bottleneck\n";
}
```
Unit tests verify correctness (*including* the empty/single-element edge cases and the variance-≥-0 *property* that would catch the Day 227 bug), and a micro-benchmark measures performance — the testing + profiling disciplines in miniature.

**Linux:** run the tests under sanitizers — `g++ -O1 -fsanitize=address,undefined test.cpp && ./a.out` (memory/UB), and for concurrent code `g++ -fsanitize=thread ...` (races). Profile with `perf record ./prog && perf report`; check cache/branches with `perf stat`. *Always* run your test suite under ASan/UBSan (and TSan for concurrent code) — they catch what tests alone miss.

## Quant Thinking Track — Verify, don't hope; measure, don't guess
The professional disciplines of reliable software: **verify correctness through testing (don't assume it), measure performance through profiling (don't guess it), and fix systematically through debugging (with the right tools).** Code is a *hypothesis*; tests and profilers are how you *verify* it. In money-handling systems, this isn't optional polish — it's how you avoid catastrophe, because the cost of an uncaught bug (minutes in a test, millions in production) makes verification the highest-leverage investment. The same intellectual honesty that demands you verify a *trading edge* (don't trust the backtest) demands you verify your *code* (don't trust that it works) — by measurement, not hope.
