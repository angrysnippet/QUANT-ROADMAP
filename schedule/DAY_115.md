## Why today matters
Searching 10⁷ values: store an index/hash (more memory, very fast) vs scan (low memory, slow). Which is better? **It depends** — that's engineering.

---

## Block 1 — Time vs space, concretely
**Find duplicates** in an array: a hash set is `O(n)` time / `O(n)` space; sorting then scanning is `O(n log n)` time / `O(1)` extra space. The "right" answer depends on whether memory or time is the binding constraint.

*Why it matters:* in production quant systems, memory limits (RAM per box, cache size) decide feasibility as often as CPU does.

---

## Block 2 — DSA · The space toolkit
- **Hashing** — buy O(1) lookups with O(n) memory.
- **Memoization** — buy speed with a table (Day 43).
- **Rolling arrays** — give back memory when only recent state matters (Day 80).
- **In-place algorithms** — O(1) extra space (e.g. two-pointer dedupe).

**Task:** for each, name when its memory cost is *worth it* and when it isn't.

## Block 3 — DSA · The caching idea (and its danger)
A cache trades memory for speed by remembering results (DP, hash maps, browser caches, market-data caches). But caches add **staleness** and **eviction** problems — a stale cached price in trading is worse than a slow correct one.

**Thinking question:** what must you guarantee before caching a value? (That it's still valid / correctly invalidated.)

---

## Block 4 — Mathematics (Quantifying the trade)
Solution A: time O(n), space O(n). Solution B: time O(n log n), space O(1). At `n = 10⁹`, O(n) space may simply not *fit* in RAM — then B wins despite being slower.

**Exercise:** 10⁹ 8-byte numbers = 8 GB — does an O(n)-space hash approach fit on a 16 GB box with overhead? (Risky.) **Challenge:** when does reducing memory *hurt* readability/maintainability?

## Block 5 — Quant Thinking (Memory is a first-class constraint)

Real quant-system decisions:
1. **Market-data storage:** tick data is enormous. You stream it (O(1) state per symbol) or memory-map it rather than loading years into RAM.
2. **Feature matrices:** an `N_assets × N_features × T` cube can blow past RAM — choose `float32` over `float64`, chunk by date, or compute online.
3. **Caching computed signals/risk:** huge speedups, but you must invalidate correctly when new data arrives — a stale risk number can be catastrophic.

**Hard puzzle:** you need the rolling covariance matrix of 500 assets over a long history, updated daily. Storing every past matrix is wasteful. What stays in memory? (Running sums `Σx`, `Σxxᵀ` and a count → covariance in O(1) update, O(k²) space — not O(T·k²).)

**Career connection:** "what's your memory footprint at scale?" and "how do you invalidate that cache?" are standard quant-dev interview probes — and daily production realities.

## Quant Thinking Track — There Is No Free Lunch
Speed, memory, accuracy, simplicity — you trade among them. The skill isn't always "use less memory" or "go faster"; it's **knowing which resource is scarce for *this* system** and spending the others to protect it.
